## 版本 1: 完整版（适合简历详细描述）

### 项目描述
基于 Olist 电商多表数据（10 万+订单、8+张表），搭建完整的数据分析体系，包括指标监控、用户分层、履约分析等模块，支持多维度业务问题定位与评估。

### 技术栈
MySQL / SQL / Tableau / Python (pandas, matplotlib)

### 核心工作

**1. 数据建模与指标体系**
- 构建星型数据模型，设计事实表（fact_orders）与 5 张主题表（dm_kpi_daily、dm_state_daily、dm_category_daily、dm_customer_rfm 等）
- 拆解 GMV 为"流量×转化×客单×复购"，沉淀 KPI 指标字典与口径说明（item_gmv vs paid_gmv），并固化到 SQL 与 BI 度量中

**2. SQL 专题分析**
- 编写 6+ 个专题分析脚本，包括：销售趋势、转化漏斗、Cohort 留存、RFM 用户分层、履约体验、差评根因分析
- 使用窗口函数、多表关联、复杂聚合实现 Cohort 留存矩阵计算，支持按首购月追踪后续月份复购率

**3. BI 看板搭建**
- 使用 Tableau 搭建 5 页经营分析看板（经营总览/增长拆解/漏斗&留存/履约&评价/RFM），实现多维切片筛选与联动分析
- 设计 Cohort 留存热力图、RFM 分层可视化、履约时长分布等核心图表，提升异常定位效率

**4. 异常归因与策略建议**
- 对转化下滑/差评上升等波动按品类/地区/卖家分层对比，联动履约时效与评分进行交叉验证
- 输出可执行优化建议：优化高风险品类履约 SLA、对高潜用户定向触达、建立差评预警机制

**5. Python 数据处理**
- 使用 pandas 进行数据质量检查（缺失值、重复值、主外键一致性）
- 验证指标口径一致性（item_gmv vs paid_gmv 差异率 < 2%）

- 交付物：SQL 脚本（6+ 专题）+ Tableau 看板（5 页截图）+ 指标文档 + 分析报告 PDF + Python Notebook

## 面试准备：关键问题与回答

### Q1: 介绍一下这个项目
**回答框架**:
- **背景**: 基于 Olist 电商公开数据集（10 万+订单），模拟真实业务场景
- **目标**: 搭建完整的数据分析体系，覆盖指标监控、用户分层、履约分析
- **技术**: MySQL 建模 + SQL 分析 + Tableau 看板 + Python 处理
- **成果**: 完整的指标体系、6+ 专题分析、3 页交互式看板、分析报告

### Q2: 你是如何设计指标体系的？
**回答要点**:
- 从业务目标出发，拆解 GMV = 流量 × 转化 × 客单 × 复购
- 定义核心指标：规模（GMV/订单/买家）、效率（客单/复购）、体验（准时率/差评率）
- 明确口径：item_gmv（经营分析）vs paid_gmv（对账），统一过滤规则（仅 delivered）
- 沉淀文档：metric_dict.md，避免口径漂移

### Q3: Cohort 留存分析是怎么做的？
**回答要点**:
- 定义首购月 Cohort：按用户首次购买月份分组
- 计算留存：追踪每个 Cohort 在后续月份的复购人数和复购率
- 使用窗口函数和多表关联实现（SQL 脚本：05_cohort_retention.sql）
- 可视化：Tableau 热力图展示留存矩阵

### Q4: 如何定位差评率上升的原因？
**回答要点**:
- 多维度下钻：按品类/地区/卖家分层对比差评率
- 关联分析：履约时长 vs 评分散点图，发现超时订单差评率高 2.3 倍
- 根因定位：差评 Top 品类/卖家，结合物流数据找到具体问题
- 输出建议：优化高风险品类履约 SLA，建立预警机制

### Q5: Tableau 看板是如何设计的？
**回答要点**:
- **Page 1 总览**: KPI 卡片 + 趋势图 + Top 排行 + 切片器（时间/地区/品类）
- **Page 2 增长**: 新老客对比 + Cohort 热力图 + RFM 分层
- **Page 3 履约**: 物流链路 + 准时率 + 评分关联 + 异常定位
- 设计原则：少而精、可交互、支持下钻

### Q6: 项目中遇到的最大挑战是什么？
**回答示例**:
- **挑战**: item_gmv 与 paid_gmv 口径不一致，差异率达 5%
- **分析**: 多支付方式、促销折扣、退款逆向流程导致
- **解决**: 明确两种口径的适用场景，沉淀到文档，并在 BI 中分别展示
- **收获**: 深刻理解口径管理的重要性
